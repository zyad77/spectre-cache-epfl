Technique d'attaque

Flush+Reload

La primitive de canal auxiliaire cache qui rend Spectre possible. Implémentation réelle annotée + simulation interactive.

Principe

Comment fonctionne Flush+Reload

Trois phases, une observation : mesurer si une adresse mémoire a été accédée par un autre processus (ou par une exécution spéculative).

F

1. FLUSH — Invalider le cache

L'attaquant vide du cache toutes les lignes du tableau de sondage avec _mm_clflush(). Après cette phase, chaque accès provoquera un cache miss coûteux (~200 cycles).

V

2. VICTIM ACCESS — Déclencher l'accès secret

Le code accède à array[secret * CACHE_LINE]. Cette ligne est chargée en cache L1/L2 (~4 cycles pour les accès suivants).

R

3. RELOAD — Mesurer les latences

On mesure via __rdtscp() le temps d'accès à chaque index. L'index avec une latence < threshold est en cache → c'est la valeur secrète.

Implémentation réelle

Code annoté — flush_reload.c

L'implémentation complète sur machine réelle (WSL/Linux, Intel x86). Chaque décision de conception est expliquée ci-dessous.

① x86intrin.h

Intrinsèques x86 pour le timing et le cache

Ce header fournit _mm_clflush(addr) (vider une ligne de cache), __rdtscp(&junk) (lire le compteur de cycles CPU de manière sérialisée), et _mm_lfence() / _mm_mfence() (barrières mémoire). Sans ces primitives hardware, l'attaque est impossible.

② CACHE_LINE = 4096

Pourquoi un stride de 4096 et non 64 ?

Une vraie ligne de cache fait 64 bytes. Le stride de 4096 (= 64 × page_size / 16) garantit que chaque entrée du probe array tombe sur une page mémoire différente, ce qui contourne les préfetcheurs matériels qui détectent les patterns d'accès réguliers sur des pages consécutives. Un stride de 512 suffirait en théorie, mais 4096 est plus robuste sur les processeurs récents.

③ Threshold = 80 cycles

Le seuil de discrimination hit / miss

En dessous de 80 cycles → cache hit (L1/L2). Au dessus → cache miss (DRAM, ~200+ cycles). Ce seuil est calibré empiriquement pour la machine cible. Il peut être ajusté via la variable d'environnement FLUSH_THRESHOLD=N ./flush_reload sans recompiler. Sur certaines machines il faut monter à 100–120 cycles.

④ Permutation mix_i

Accès dans un ordre pseudo-aléatoire pour contrer le préfetcheur

mix_i = ((i * 167) + 13) & 255 génère une permutation de 0..255 à partir de la suite i = 0..255. En accédant aux entrées dans un ordre non séquentiel, on empêche le prefetcher matériel (qui détecte les strides réguliers) de charger "en avance" des lignes qui fausseraient la mesure. 167 est choisi copremier avec 256.

⑤ filtered_guess (sans index 0)

Pourquoi ignorer l'index 0 ?

L'index 0 (array[0]) est souvent faussement chaud dans le cache pour des raisons indépendantes de l'attaque : préchargement par le loader, accès lors du memset, etc. En reportant séparément le meilleur score parmi les index 1..255, on obtient une estimation plus fiable dans les cas où le secret vaut 0 ou quand index 0 est un faux positif systématique.

Analyse

Pourquoi ça marche

⏱

Différence de timing mesurable

L1 hit : ~24 cycles (mesuré). DRAM miss : ~200 cycles. Threshold à 80 cycles. La séparation est nette et stable sur Intel.

📐

Stride 4096 bytes

Chaque entrée du probe array est sur une page mémoire différente. Contourne les préfetcheurs qui opèrent sur des patterns intra-page. Plus robuste que le stride 512 habituel.

🔄

1000 itérations + scores

On répète 1000 fois et on agrège les compteurs scores[i]. Un signal à 98%+ émerge clairement du bruit même avec des interruptions système ponctuelles.

🔀

Permutation anti-préfetch

L'accès dans l'ordre mix_i = (i×167+13)&255 casse le pattern séquentiel que le préfetcheur détecterait et exploiterait, faussant les mesures.
